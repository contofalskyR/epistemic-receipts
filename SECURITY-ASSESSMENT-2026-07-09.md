# Security Assessment — 2026-07-09

Independent follow-up to `SECURITY-REVIEW-2026-06-12.md`. Scope: everything that changed or shipped in the four weeks since that review, plus a fresh check that its invariants (documented in AGENTS.md under "security-model") still hold, now that the GitHub repo itself is confirmed **public** (verified via unauthenticated `raw.githubusercontent.com` fetch). Read-only review — a separate agent session had uncommitted changes in the working tree throughout; those six files were diff-reviewed, not modified, and nothing in this repo was edited as part of this assessment.

**Method:** manual review of middleware.ts, lib/adminAuth.ts, app/api/login/route.ts, next.config.ts, vercel.json, all 11 `.github/workflows/*.yml`; three parallel targeted sweeps (route-auth invariants, secrets/raw-SQL, dependencies/headers/CI); `npm audit`; `git log`/`git show` for history questions. Everything below is cited to a file and line, and the two highest-severity findings (#1, #3/#4) plus the npm audit numbers were independently re-verified by direct file reads and a fresh `npm audit` run before finalizing this doc.

## Verdict

The June 12 hardening pass holds up for what it covered — timing-safe admin/login comparisons, fail-closed `ADMIN_TOKEN` defaults, correct cookie flags, no raw-SQL string interpolation of user input anywhere in the current tree, and correct session+ownership checks on the newer collections/alerts/litigation routes. Nothing from that review has regressed.

What's new is worse than what it replaced. In the four weeks since, this repo picked up a **live, working credential committed in plaintext** (not caught before push despite gitleaks running in CI since three days prior), **two cron routes that fail open** instead of closed, a **critical RCE-class dependency** added to the production bundle, and the repo going publicly readable turned the June review's hypothetical "if you open-source this, trim CONSULTANT.md" warning into a live, unaddressed exposure. None of this is exotic — it's the ordinary entropy of fast iteration outrunning a point-in-time review, which is exactly why a repo like this needs the invariants enforced continuously (CI gate + branch protection), not just re-verified periodically.

## Patch plan

**Do first, outside of code (Robert only — I can't do this for you):**
1. Rotate the Telegram bot token now (BotFather → `/revoke`, issue a new one) — finding #1. Treat it as already compromised; rotation matters regardless of whether history gets scrubbed. Update every consumer to read from `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID` env vars instead (see P0 below).
2. Check GitHub → Actions history for the `6366078` push, and Settings → Branches for `main`'s protection rules, to close the loop on finding #16.

**P0 — small, mechanical, safe once the other agent's session clears:**
- Flip the two fail-open cron checks to fail-closed (#3, #4): `if (!cronSecret || header !== cronSecret) return 401` — mirror whatever pattern `/api/ingest/scotus` already uses correctly.
- Replace hardcoded `BOT_TOKEN`/`CHAT_ID` in `scripts/notify-telegram.sh` and the four `OWNER_CHAT_ID = "7688025079"` literals (`app/api/books/[bookId]/request-analysis/route.ts:7`, `app/api/feedback/route.ts:5`, `app/api/search/miss/route.ts:5`, `scripts/generate-reasons.ts:87`) with env vars, matching the pattern `app/api/cron/topic-alerts/route.ts:8` already uses correctly (#1, part of #7).
- `git rm --cached app/opinions/OpinionsClient.tsx.bak` (#12 — already on record as not-done).
- `npm audit fix` (non-breaking fixes) and evaluate a path off `@xenova/transformers` — or at minimum confirm the ONNX model-loading path never touches attacker-controlled bytes at runtime — and check for a patched `@sentry/nextjs`/`uuid` release (#2, #6).

**P1 — needs a bit more judgment/testing:**
- Add real `auth()` + org-ownership checks to the Stripe portal/checkout routes before they're wired up for real users (#5).
- Enforce `expiresAt` in `verifyApiKey` (#8).
- Tighten CSP `connect-src` to an explicit allowlist instead of `https:` (#9).
- Fix the script-injection shape in `notify-telegram.yml` and the 5 similar workflows — route event data through `env:` (#10).
- Pin `gitleaks/gitleaks-action`, `actions/checkout`, `actions/setup-node` to commit SHAs (#11).

**P2 — hygiene/process, not urgent:**
- Decide whether `CONSULTANT.md` and similar planning docs belong in a public repo at all; if not, `git rm --cached` + gitignore rather than leaving them tracked (#7).
- De-duplicate the `isAdmin()` reimplementation in `app/api/admin/api-keys/*` (#13).
- Add a runtime assertion backing `lib/effective-axis.ts`'s identifier-safety comment (#14).
- Add `preload` to HSTS once ready to submit to the preload list (#15).
- Add a local pre-commit/pre-push secret scan (gitleaks has a git-hook mode) so a leak like #1 can't reach `main` at all, independent of what CI catches after the fact (#16).
