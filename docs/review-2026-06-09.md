## 1. Inconsistencies

### Security (fix before anything public)

1. **Hardcoded API keys committed.** `scripts/enrich-media-coverage.ts:13` (NYT key), `scripts/ingest-colombia-legislation.ts:19` (Azure Search key). Rotate both, move to env. Since the repo is private these survive, but they're in git history — rotation, not just deletion.
2. **Cron auth bypass.** `app/api/cron/topic-alerts/route.ts` checks `if (cronSecret && ...)` — if `CRON_SECRET` is unset, the endpoint is open. The SCOTUS route (`app/api/ingest/scotus/route.ts:78`) does it correctly (strict compare). Make them consistent.
3. **Unguarded spend + write endpoints.** `app/api/books/upload/route.ts` (DB writes, no size cap on uploaded PDFs) and `app/api/books/[bookId]/ingest/route.ts` (spends `ANTHROPIC_API_KEY`) have no `isReadOnly()` gate, unlike the match route. Today the sitewide `SITE_PASSWORD` hides this; the moment that's removed, anyone can upload books and burn your API budget.
4. **Personal Telegram chat ID hardcoded** in 3 places (`notify-telegram.yml`, `topic-alerts/route.ts`, `match-book-to-graph.ts`). Env var.
5. **Neon pooler hostname committed** in `AUDIT-2026-05-21.md`. Cosmetic, but scrub before any repo-opening.

### Status-field sprawl (the biggest conceptual inconsistency)

`Claim` carries **four** overlapping status fields: `currentStatus` (deprecated, still defaulting new rows to `"DISPUTED"`), `verificationStatus`, `epistemicStatus`, `epistemicAxis`. The repo's own diagnostic shows the damage: 943 rows with `HARD_FACT` inside `verificationStatus`, 2,011 with `VERIFIED` inside `currentStatus`, 86.9% NULL `epistemicStatus`, plus undocumented values (`established`).

Compounding it, every document disagrees on the vocabulary:

| Source | Status model |
|---|---|
| Your project description | 4-way axis: SETTLED / CONTESTED / RECORDED / OPEN |
| `schema.prisma` + `/api/search` | 5-way: adds UNRESOLVABLE |
| `WHITEPAPER.md` | HARD_FACT / PROVISIONAL / DISPUTED / DEPRECATED |
| `currentStatus` docstring | DISPUTED / HARD_FACT / NEVER_RESOLVES / VERIFIED |

Recommendation: declare `epistemicAxis` + `verificationStatus` canonical, convert both to **Prisma enums** (almost every typed field in the schema is a free string — that's how `HARD_FACT` leaked into the wrong column), stop writing `currentStatus`/`epistemicStatus`, drop the `"DISPUTED"` default, and make `epistemicAxis` required after backfill. Update WHITEPAPER to match.

### Three relation vocabularies

- `Edge.type`: FOR / AGAINST / CITES / RETRACTS / CORRECTED
- `ClaimRelation.relationType`: lowercase `cites`/`cited_by`/`related` mixed with uppercase OUTCOME / SUPERSEDED_BY / REVERSED…
- `BookClaimMatch.matchType`: SUPPORTS / CONTRADICTS / RELATED / UNVERIFIED

Your own project description says ClaimRelation holds "SUPPORTS/CONTRADICTS" — it doesn't; that's BookClaimMatch. When the owner's mental model and the schema diverge, an API consumer has no chance. Unify casing and publish one relation ontology (this becomes your API contract anyway).

### Documentation drift (four different claim counts in-repo)

- `AGENTS.md` registry: "~336,900 claims, 90+ pipelines" (synced 05-21)
- `WHITEPAPER.md`: "~141,900 claims (May 2026)"
- `docs/epistemic-status-diagnostic`: 1,466,486 (06-08)
- `docs/investor-memo`: "1,534,209 across 176 pipelines"

The AGENTS.md registry is the file your agents read — it's 4–5x stale and contains entries the prior audit already flagged as wrong (`icd11_v1 = 0` vs 1,374 in DB). **Generate the registry from the DB** (a script that writes the markdown table from `GROUP BY ingestedBy`) instead of hand-editing. Same for WHITEPAPER stats: stamp "figures auto-generated YYYY-MM-DD".

### Repo hygiene

- ~150 log files and dry-run JSONs at repo root (`.pipeline-*.log`, `pipeline-NN-dry-run-sample.json`, `enrich-*.log`), `CONSULTANT.md` at 581 KB, cursor-state files (`.nara-cursor.json`). Move to `artifacts/` and gitignore; root is unreadable right now.
- `README.md` is still the stock create-next-app template. For a project courting researchers/investors, the README is the front door — the investor memo is better positioned than the repo itself.
- Two migrations share the timestamp `20260608000000` (add_epistemic_axis, add_watched_topic) — ordering is ambiguous on a fresh `migrate reset`/shadow DB. Rename one.
- 339 scripts with no tests anywhere and no CI for lint/typecheck (workflows only do ingestion). Given Pipeline 5's fabrication incident, even a thin test layer over shared parsing/validation helpers would pay off.
- 26 hardcoded academic-field directories (`app/physics`, `app/mathematics/data.ts|data2.ts|data3.ts`, …) with per-page data files; `earth-sciences` and `geology` both exist. Consolidate into a dynamic `app/fields/[slug]` route reading from `AcademicField`/`Topic` tables — content in checked-in `.ts` files is exactly the "static" pattern you want to escape.

### Unimplemented "living" machinery

The whitepaper's core promise (§"self-auditing") has schema but no runtime: `AiJob` ("no worker in v1"), `SuggestedThresholdEvent` and `SourceCredibilityEvent` empty (per your audit). Total automation today: **2 Vercel crons (SCOTUS daily, topic-alerts weekly) + 1 scheduled GitHub Action (NYT coverage)** — out of ~176 pipelines. Everything else is a manual laptop run. This is the gap between the pitch ("changes its mind when evidence does") and the system (changes its mind when Robert runs a script).

### Smaller code flags

- `app/api/search/route.ts` uses Prisma `contains` (ILIKE) — fine with your trgm index, but `export const revalidate = 60` very likely does nothing on a dynamic route handler that reads query params; verify, and cache explicitly if intended.
- Middleware rate limiting is per-edge-isolate in-memory — effectively decorative under real load or a multi-region deployment.
- `match-book-to-graph.ts` judge errors `resolve([])` silently — failed judgments are indistinguishable from "no matches"; the error counter never increments.
- AUDIT items still open as of this zip: uspto_v1 edges not cascaded, `retraction_watch_v1` orphan records, nobel_v1 39% deprecated without documentation.

## 2. Book pipeline (your flagged issue is worse than described)

The judge step doesn't "default to Sonnet" — it shells out `exec('claude --print …')` with **no `--model` flag at all** (`match-book-to-graph.ts:162`), so it uses whatever the CLI default is, unpinnable per-run. And `app/api/books/[bookId]/match/route.ts` spawns a **detached `npx ts-node` child** that calls that CLI — this cannot run on Vercel (no persistent processes, no CLI, ephemeral FS, file-based progress tracking). The book feature is structurally laptop-only.

Fix in order:
1. Replace CLI shell-outs with `@anthropic-ai/sdk` (already used correctly with pinned Haiku in the ingest route — `app/api/books/[bookId]/ingest/route.ts:15`).
2. Use the **Message Batches API** for the judge step: 50% off, perfect fit for thousands of independent judgments per book.
3. Move the job to a real background-job system (Inngest / Trigger.dev / QStash), progress in a `BookMatchJob` DB row instead of files.
4. Candidate retrieval: replace keyword FTS with **pgvector** (Neon supports it). One-time embedding of 1.6M claims is cheap; then ANN top-k + Haiku judge only on the top candidates. Cuts judge calls dramatically and finds matches keywords miss.

## 3. Missing data

Two different gaps — internal depth and external breadth. Internal first, because your own investor memo names it the #1 risk:

### Internal (depth) — the empty receipts
- Median bulk claim: 0 sources, 0 edges, axis from a single pipeline. Per your audit: ~49% of `nih_reporter_v1` sources missing `url`; several pipelines 100% NULL `publishedAt`. For a provenance product, **backfilling source URLs/edges on existing claims is worth more than any new pipeline.**
- Only ~11.3K CONTRADICTS edges, nearly all retraction-driven. No systematic contradiction mining between existing claims (you have `detect-contradictions.ts` — schedule it).
- No `Claim.updatedAt` / `lastVerifiedAt` — you cannot answer "when did we last check this is still true?", which is the literal product promise.
- No embeddings — blocks semantic search, related-claims, dedup, and book matching quality.

### External (breadth) — ranked by fit to "contested claims with receipts"
1. **Fact-checked/contested claims corpus** — Google Fact Check Tools API (ClaimReview markup from IFCN fact-checkers). Your memo admits "do vaccines cause autism" returns nothing; this is the highest-demand query class and the namesake of the product. Also: Wikipedia "list of controversies" seeds, RetractionWatch reason taxonomy.
2. **US state legislation** — OpenStates/Plural API (50 states). You have ~40 national parliaments but no US state statutes; state supreme courts are already scripted (`ingest-courtlistener-state-supreme.ts`), legislatures are the missing half.
3. **Election results + public opinion beyond CCES** — MIT Election Lab, ANES, Pew/Gallup toplines; your FEC coverage is thin (600 + 600 claims, 3 cycles).
4. **Economic series** — FRED is scripted but never run (key pending per AGENTS.md); add BLS, BEA, Census ACS, Eurostat, IMF WEO, OECD. These give CONTESTED political claims a SETTLED numeric anchor to link against.
5. **Regulatory beyond FDA** — EMA (EU drug approvals — natural pair to your FDA spine), EPA ECHO enforcement, executive orders + the full Federal Register (you only ingest "significant final rules").
6. **Clinical evidence above the trial level** — ClinicalTrials.gov *results* sections (you have registrations), Cochrane reviews, USPSTF recommendations. "Approved" is recorded; "does it work" is the contested layer.
7. **Climate/environment** — NASA GISTEMP/NOAA series, IPCC AR6 assessed statements (high contestation value, clean provenance).
8. **UN General Assembly votes** (Voeten dataset) — you have UNSC resolutions; UNGA roll-calls extend the Voteview pattern globally.
9. **Preprints + replication** — OSF registrations, Institute for Replication outcomes; lets you mark claims "replicated/failed-to-replicate" — a status no competitor tracks.
10. **Patents, done right** — PatentsView bulk API to replace the fabricated `uspto_v1` (the failure was methodology, not the domain).
11. **Media assertion layer** — GDELT/MediaCloud to record *who asserted what when*. BillCoverage is NYT-only; "dark-matter bills" needs multi-outlet data. This is also what the policy-intelligence wedge sells.

### Schema gaps for the above
`metadata Json` on Source/Edge/MetaEdge (already queued), a `PipelineRun` table (pipeline, startedAt, rowsWritten, cursor, status), unique constraint on `BookClaimMatch(bookClaimId, claimId)`, and a `ClaimStatusHistory` table so axis flips are themselves receipted.

## 4. Launching public + live

### Phase 0 — Pre-launch hardening (~1 week)
- Rotate the two committed keys; env-var the Telegram ID; strict `CRON_SECRET` everywhere.
- Gate every write/spend route behind auth (`isReadOnly()` at minimum); add upload size caps.
- Replace in-memory rate limiting with Upstash Redis (or Vercel WAF rules).
- Keep `SITE_PASSWORD` only for `/admin` + review routes; open the read paths.
- Real README; registry auto-generated from DB; move logs/dry-runs out of root.

### Phase 1 — The freshness backbone (2–4 weeks) — this is what makes it "live"
- **`PipelineRun` table + public `/status` page**: per-dataset lastSyncedAt, rows added, source lag. "Living graph" is only credible if freshness is *visible*. Stamp every claim page with "last verified".
- **Tier pipelines by cadence** and schedule them — GitHub Actions cron is free and you already have the pattern (`refresh-bill-coverage.yml`):
  - 15 min–hourly: USGS feed, GDELT (if added)
  - Daily: CrossRef retractions, Federal Register, CourtListener (extend past SCOTUS to circuits), openFDA, OFAC, congress.gov votes (replaces static Voteview file going forward)
  - Weekly: OpenAlex incremental (`from_updated_date` cursor — you currently have a one-shot snapshot)
  - Monthly/quarterly: World Bank, WHO GHO, SIPRI, UCDP
- **Wire the event loop you already designed**: new retraction → CONTRADICTS edge → axis flip → `ThresholdEvent` → appears in `/feed` + RSS/Atom per topic + email (Resend + `TopicSubscription` already exist) + Telegram. The retraction→flip behavior is your crown jewel; today it only happens when you run a script. RSS feeds are the cheapest possible "this thing is alive" signal.
- Long-running ingests (OpenAlex, books) on a small always-on worker (Fly/Railway/Hetzner) or Inngest — not Vercel functions.

### Phase 2 — Public read API (4–8 weeks) — liveness as product
- v1 read-only endpoints: `GET /v1/claims/{id}/receipt`, `/v1/search`, `/v1/retractions?since=`, `/v1/verify?statement=` (the opus-brainstorm's "ship the retraction feed first" is right — lowest effort, clearest buyer).
- API keys + metering (Unkey or hand-rolled + Stripe), generous free tier for researchers; this is the prerequisite for every monetization path you listed, and nothing in the repo supports it today (the only auth primitive is one shared password + `Profile.key`).
- Webhooks for subscribed events ("claim X changed axis") — the live feature B2B customers actually pay for.
- OpenAPI doc + a public "method" page (the neutrality asset).

### Phase 3 — Live to humans
- Real auth (Clerk/Auth.js) replacing `Profile.key`; saved searches → alert emails (plumbing exists).
- SEO: sharded sitemaps for 1.6M claim pages + ISR + `schema.org/ClaimReview` structured data + per-claim OG "receipt card" images. This is how a claims DB gets discovered; right now the site is password-walled and invisible.
- Corrections/flagging workflow surfaced publicly (`/corrections` exists) — community signals become `SourceCredibilityEvent` producers, filling the empty table.
- Semantic search + "related claims" from the Phase-1 embeddings.

### The metric that ties it together
Track and publish three numbers on `/status`: **% of claims with ≥1 source URL and ≥1 edge** (receipt completeness), **median pipeline freshness lag**, **% claims with non-NULL `epistemicAxis`**. Those three are the honest difference between "1.6M rows" and "1.6M receipts".

## 6. How to hand this to the agent

Drop this file into the repo as `docs/review-2026-06-09.md` and feed it in tiers — not all at once. A 20-item dump gets partial, sloppy execution.

**Prompt per tier, e.g.:** "Read docs/review-2026-06-09.md. Execute P0 items 2–5 (skip key rotation — I'll do that), one commit per item, show me the diff before pushing."

**Items the agent can't do — these are yours:**
- #1: rotating the NYT/Azure keys in the provider dashboards
- #2/#4: setting `CRON_SECRET` and `TELEGRAM_CHAT_ID` in Vercel env settings
- #6: the 4-way vs 5-way axis decision — editorial call, decide before the agent touches the migration
- Phase 2 billing accounts (Upstash, Unkey/Stripe), and pending API keys (FRED, IUCN)

**Items the agent must not run unsupervised:**
- #6 (enum migration) — touches 1.6M rows. Require dry-run row counts and the full SQL printed for approval before applying, per your own AGENTS.md "verify against DB state" rule.
- #9 (migration rename) — affects every environment that has applied migrations; review the `migrate resolve` plan first.
- #20's `updateMany` — have it print the matched count before executing.

**Suggested order:** P0 (one session, verify each) → #9, #11, #13 (low-risk hygiene) → #6 with your review → remaining P1 → P2 alongside Phase 1 of the launch plan.
